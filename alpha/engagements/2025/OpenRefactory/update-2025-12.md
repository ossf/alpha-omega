# Entry for the Annual Report

OpenRefactory worked in collaboration with Piotr Karwasz of Apache Software Foundation to unlock the capability to generate Vulnerability Exploitability eXchange (VEX) documents automatically for the Java ecosystem.

The collaborative effort resulted in open sourcing several capabilities:

(a) Generating static call graphs analyzing the source code of Java programs,
(b) Identifying the root cause functions/methods for each CVE, and
(c) using the call graphs and the root cause information to generate VEX documents.

A reference GitHub Actions based workflow was created that demonstrated how the components are to be used in the CI/CD pipeline. A demo video of the workflow in action is available at: https://www.youtube.com/watch?v=93iEw8LyeIc

The team specifically worked with Apache Solr to generate VEX documents for their 65 outstanding CVEs in the dependencies. But during the course of the project, they also generated VEX documents and root cause information for Apache Hadoop, Apache Kafka, Apache Commons Compress, Apache Commons Text and Apache Commons Configuration.

One of the major enhancements proposed for VEX is the introduction of VEXplanations. Today’s VEX documents provide only a single machine-readable status (“affected” / “unaffected”) accompanied by a human-readable note. However, a recent Open Source Economy survey on VEX usage found that the lack of verifiable justification is the top concern among VEX consumers. VEXplanations address this gap by supplying a detailed, machine-readable rationale describing why a vulnerability is considered reachable or not. This additional context enables users to independently verify the correctness of the VEX assessment.

The work was featured in Apache Software Foundation's 2025 year in review as one of the key highlights.
