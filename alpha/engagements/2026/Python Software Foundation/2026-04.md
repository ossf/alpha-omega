## Security Developer-in-Residence (Seth Larson)

### Python Security Response Team

* Published 11 advisories with patches for CPython and pip just in April. For reference,
  there were 21 advisories published in the entirety of 2025 :')
  * [CVE-2026-3087] shutil.unpack_archive() doesn't check for Windows absolute paths in ZIPs
  * [CVE-2026-3298] Out-of-bounds write in Windows asyncio.ProacterEventLoop.sock_recvfrom_into() when using nbytes
  * [CVE-2026-3219] pip doesn't reject concatenated ZIP and tar archives
  * [CVE-2026-5713] Out-of-bounds read/write during remote debugging when connecting to malicious target
  * [CVE-2026-4786] Incomplete mitigation of CVE-2026-4519, %action expansion for command injection to webbrowser.open()
  * [CVE-2026-6100] Use-after-free in lzma.LZMADecompressor, bz2.BZ2Decompressor, and gzip.GzipFile after re-use under memory pressure
  * [CVE-2026-3446] Base64 decoding stops at first padded quad by default]
  * [CVE-2026-1502] HTTP client proxy tunnel headers not validated for CR/LF
  * [CVE-2026-5271] Python install manager script aliases search path hijack
* Created `ghsa-cli` tool which patches some of the shortcomings of
  GitHub Security Advisories such as quickly adding collaborators,
  crediting people for coordinating, developing or reviewing remediations,
  or creating CVE records from a GHSA ticket. GitHub:
  https://github.com/sethmlarson/ghsa-cli and published to PyPI: https://pypi.org/project/ghsa-cli
* Began crediting all contributors to advisories in the CVE record with a link
  to their name and GitHub page. This is a part of "Open Source Security isn't Special"
  and making security work more like regular Open Source contribution through
  recognition of work.
* Added feature to PSRT GHSA bot so that non-admin security response team
  members can close tickets by adding `[CLOSED]` to the title.
* Fixed pagination in PSRT GHSA bot.

## PyPI Safety & Security Engineer (Mike Fiedler)

### Malware Response

In April, the PyPI security inbox processed roughly 420 conversations,
with approximately 350 confirmed malware reports.
The remainder were false positives or administrative messages about open vulnerabilities.
Auto-quarantine continued to fire reliably, taking down many malicious projects within minutes of upload.

Mike co-authored an [incident report blog post with Seth Larson](https://blog.pypi.org/posts/2026-04-02-incident-report-litellm-telnyx-supply-chain-attack/)
on the recent `litellm` and `telnyx` supply-chain compromises,
with practical guidance for Python developers and maintainers
on what to do when (not if) the next one lands.

Mike also worked alongside several maintainers whose projects were compromised in April —
through stolen tokens, account takeovers, and multi-account campaigns.
He helped them coordinate cleanups, publish advisories, restore project state,
and adopt stronger publishing controls (Trusted Publishing, Organizations, 2FA hardening).
A few of these investigations are still open
and have already fed back into PyPI's detection rules.

### Trail of Bits Security Audit

Mike published "[PyPI Completes its Second Security Audit](https://blog.pypi.org/posts/2026-04-16-pypi-completes-second-audit/)"
on April 16th (also [echoed on the Alpha-Omega blog](https://alpha-omega.dev/blog/the-python-package-index-has-completed-its-second-audit/)),
summarizing the audit conducted by [Trail of Bits](https://www.trailofbits.com/)
and funded by the [Sovereign Tech Agency](https://www.sovereign.tech/).
The [full audit report](https://raw.githubusercontent.com/trailofbits/publications/beb3c749f8398473682a6a7391be89c1882390a9/reviews/2026-04-pypi-warehouse-securityreview.pdf)
is published on the Trail of Bits publications page.

Of the 14 findings (2 High, 1 Medium, 7 Low, 3 Informational),
12 were remediated and 2 were explicitly accepted with documented rationale in the blog post.
April work shipped a fix for the second High-severity finding,
[TOB-PYPI26-13](https://github.com/pypi/warehouse/pull/19749),
where stale records could silently grant cross-organization project access,
plus [TOB-PYPI26-14: improved admin logging](https://github.com/pypi/warehouse/pull/19751).
The audit also produced custom CodeQL queries
that Mike has been [integrating into PyPI's CI pipeline](https://github.com/pypi/warehouse/pull/19925)
to catch similar bug classes,
plus proposal reviews for upcoming features (per-organization Trusted Publishers, TOTP hardening).

### Trusted Publishing and Account Associations

Mike continued the GitLab Self-Managed (GLSM) Trusted Publishing rollout —
issuer URL handling, per-issuer JWKS storage, pending-publisher reification —
and started lining up beta testers.

In response to a reported pending-publisher hijack scenario,
Mike shipped three defense-in-depth changes: [notifications on pending-publisher reify](https://github.com/pypi/warehouse/pull/19971),
so users learn immediately when a pending publisher they configured has been claimed;
[surfacing pending-publisher conflicts](https://github.com/pypi/warehouse/pull/19972) instead of silently redirecting;
and [time-based expiration of unused pending publishers](https://github.com/pypi/warehouse/pull/19996).

Mike also shipped [project permission checks before email/2FA validation on upload](https://github.com/pypi/warehouse/pull/19952),
[blocked Trusted Publishing uploads to quarantined projects](https://github.com/pypi/warehouse/pull/19860),
and added a [confirmation prompt before removing Trusted Publishers](https://github.com/pypi/warehouse/pull/19001).

### Security Infra Improvements

#### Upload-path hardening

Mike shipped [rejection of ZIP archives with mismatching file sizes](https://github.com/pypi/warehouse/pull/19777)
and additional [spoofed ZIP handling](https://github.com/pypi/warehouse/pull/19815),
closing a previously-reported YARA-scanner bypass
and resolving a coordinated-disclosure case.
He also added [YARA-based rejection of `sourcedefender`-obfuscated uploads](https://github.com/pypi/warehouse/pull/19927)
to shut another common obfuscation channel.

#### Admin tooling

[Server-side pagination for admin observation listings](https://github.com/pypi/warehouse/pull/19914)
keeps the admin UI responsive on large datasets,
and [release vulnerability data is now surfaced in the admin UI](https://github.com/pypi/warehouse/issues/19916)
for faster triage.

#### Performance and reliability

Cache-purge work cut [unnecessary purging](https://github.com/pypi/warehouse/pull/19898),
added [purge-reason logging](https://github.com/pypi/warehouse/pull/19861),
and removed redundant flush calls.
Mike also fixed an intermittent [UniqueViolation issue](https://github.com/pypi/warehouse/pull/19844)
and an [n+1 query on the account publishing page](https://github.com/pypi/warehouse/pull/19934).

#### CI/CD and hygiene

Mike adopted [`zizmor`](https://github.com/pypi/inspector/pull/280)
to harden GitHub Actions workflows in `pypi/inspector`,
[excluded malware submissions from search indexes](https://github.com/pypi/warehouse/pull/19875),
shipped a [callout discouraging mass-scan "security research"](https://github.com/pypi/warehouse/pull/19855)
on the registration flow,
and continued blocking additional [disposable email domains](https://github.com/disposable-email-domains/disposable-email-domains/pull/944).
