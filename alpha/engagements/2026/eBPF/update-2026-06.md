# eBPF Foundation Alpha-Omega Engagement

## Project Update — June 2026

This update covers progress since the previous Alpha-Omega project report and focuses on ongoing work to add KASAN (Kernel Address Sanitizer) support for JITed eBPF programs.

## Advancing KASAN Support for JITed eBPF Programs

The KASAN workstream has continued to progress from proof-of-concept implementation toward an upstream-ready feature for the Linux kernel.

KASAN is a kernel debugging and hardening technology that detects memory safety issues such as out-of-bounds accesses and use-after-free bugs. Extending KASAN coverage to JITed eBPF programs improves the ability to detect memory errors in a critical execution path used by observability, networking, and security workloads.

### RFC Review and Community Engagement

A major milestone during this reporting period was the publication and review of the initial RFC implementation.

Feedback from kernel maintainers and the broader BPF community helped identify several areas for refinement, including instruction filtering, stack-access handling, test coverage, and maintainability of the implementation. The project team incorporated review feedback and continued iterating on the design.

The work was also presented at the Linux Storage, Filesystem, Memory Management, and BPF Summit (LSFMM/BPF), where discussions provided additional validation of the overall approach and surfaced useful feedback on instruction coverage, verifier integration, and long-term maintainability.

### Expanded Instruction Coverage

Since the previous update, support has been expanded beyond the initial load/store implementation.

Work has focused on:

* support for BPF atomic operations;
* handling of `BPF_ST` and `BPF_LD` instructions in addition to `BPF_STX` and `BPF_LDX`;
* refinement of handling for load-acquire and store-release operations;
* improved differentiation between stack and non-stack memory accesses; and
* continued work to ensure instrumentation is only applied where appropriate.

This effort required deeper analysis of how various eBPF instruction classes are translated by architecture-specific JIT compilers, particularly on x86-64.

### Improved Testing Infrastructure

A significant amount of effort during this reporting period was dedicated to testing and validation.

The original proof-of-concept tests were expanded into a more comprehensive framework capable of validating:

* different instruction types;
* varying access sizes;
* read and write operations;
* use-after-free scenarios;
* out-of-bounds accesses; and
* cases that should not trigger KASAN instrumentation, such as verifier-approved stack accesses.

The testing framework was also enhanced to programmatically detect KASAN reports and validate expected behavior, reducing reliance on manual log inspection.

### Architecture and Runtime Fixes

As implementation matured, several issues were identified and resolved involving:

* verifier-to-JIT metadata handling;
* stack-access classification;
* instruction offset calculations;
* interactions with `BPF_PROBE_MEM` handling;
* call-depth accounting on x86-64;
* JIT test compatibility when KASAN instrumentation is enabled; and
* stability issues uncovered during extensive self-test execution.

Additional debugging and validation work helped eliminate crashes encountered during early implementation and improved confidence in the overall approach.

### Instrumentation Metrics and Observability

To better understand the impact of KASAN instrumentation, experimental observability mechanisms were added during development, including debug statistics tracking:

* number of generated KASAN checks; and
* total JIT-generated bytes associated with KASAN instrumentation.

These measurements are helping evaluate the runtime and code-size impact of the feature as development continues.

## Current Focus

The project is currently focused on:

* completing remaining review items identified during RFC discussion;
* finalizing support across all targeted instruction classes;
* continuing expansion of self-test coverage;
* refining verifier-assisted instruction filtering;
* reducing implementation complexity where possible; and
* preparing an upstream-ready version of the patch series.
