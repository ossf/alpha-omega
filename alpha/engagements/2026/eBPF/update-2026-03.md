A great deal of progress has been made in the first phase of the eBPF Foundation's 2026 Alpha-Omega engagement. Work during this period centered on two complementary tracks: an external security assessment of core eBPF execution paths and upstream engineering work to improve KASAN coverage for JITed eBPF programs.

## Security Assessment Progress

The external assessment completed an in-depth review of the x86-64 and arm64 eBPF JIT compilers and expanded to include riscv64, together with focused analysis of JIT-verifier integration assumptions.

Across this work, the review found that the JIT implementations themselves are relatively small, mechanically structured, and heavily dependent on verifier-enforced invariants. No standalone security vulnerabilities were identified in the x86-64, arm64, or riscv64 JIT backends.

The more significant results came from the verifier and JIT-verifier interaction analysis. A total of 14 findings were identified: 8 high-risk issues, 2 medium-risk issues, and 4 low-risk issues. These findings include verifier bypass vulnerabilities, memory and pointer leak issues, a race condition that can lead to memory corruption, and a denial-of-service issue affecting verifier behavior.

Two low-risk verifier issues have already been fixed upstream. The remaining findings continue through disclosure and remediation workflows.

A key takeaway from this phase is that eBPF security depends heavily on verifier correctness. Because all JIT backends rely on verifier-enforced invariants, flaws in verifier state tracking, path pruning, scalar bounds propagation, and related logic can have security impact even when the JIT compilers themselves appear sound.

## eBPF JIT KASAN Hardening Work

In parallel with the assessment, implementation work began to improve KASAN coverage for eBPF JIT programs, with an initial focus on x86-64 and generic KASAN.

Early work concentrated on understanding how eBPF JIT memory is allocated, how shadow memory is managed across architectures, and which memory regions require special handling, including private stacks, helper-returned values, and map-backed data.

The project then moved from analysis into prototyping. A proof-of-concept test setup was developed using `bpf_testmod` kfuncs that return invalid or freed memory so that JITed eBPF programs can be driven into controlled bad-access scenarios. This helped validate the relevant instruction paths, especially `BPF_LD`, `BPF_LDX`, `BPF_ST`, and `BPF_STX`, and confirmed that KASAN instrumentation work should focus on the JIT-emitted load and store paths.

From there, implementation work started in earnest. An RFC was prepared and sent to the BPF mailing list, a new Kconfig gating model was introduced for the feature on x86-64, and initial JIT-side plumbing was added to support direct calls to `__asan_loadX` and `__asan_storeX`.

Additional verifier and aux-data changes were also explored so the JIT can identify which instructions access stack memory and selectively emit checks. Current work is centered on debugging early boot and runtime issues, refining stack-access instrumentation, and better understanding red-zone and poisoning behavior so the first upstreamable version can be scoped correctly.

## Coordination and Next Steps

This Alpha-Omega work is already producing value in two ways: first, by surfacing concrete, validated verifier vulnerabilities for upstream remediation; and second, by laying the groundwork for stronger runtime diagnostics in JITed eBPF programs through KASAN integration.

The next phase is expected to continue verifier-focused analysis and remediation follow-up while advancing the JIT KASAN prototype toward a cleaner, upstream-ready implementation.
