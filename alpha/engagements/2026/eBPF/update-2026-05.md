# eBPF Foundation Alpha-Omega Engagement

## Project Update — Mid 2026

The eBPF Foundation’s Alpha-Omega engagement continues to advance two complementary security workstreams:

1. Independent security assessment of eBPF verifier and JIT behavior
2. Upstream hardening work to improve KASAN coverage for JITed eBPF programs

This update covers progress since the previous public report.

## Security Assessment Progress

Following the initial audit and disclosure process, remediation and validation work has continued on the verifier-related findings identified during the assessment.

The earlier review established that no standalone vulnerabilities were identified in the x86-64, arm64, or riscv64 JIT implementations themselves. The primary focus of follow-on work has therefore remained on verifier assumptions, verifier/JIT interactions, and related edge cases.

Additional effort during this reporting period has focused on:
- coordinating disclosure and remediation activities;
- validating fixes for previously identified findings;
- refining reproducer and testing workflows; and
- continuing analysis of verifier state handling and edge-case behavior.

The engagement continues to reinforce an important architectural lesson for the ecosystem: the security of eBPF execution depends heavily on verifier correctness and the invariants shared between verifier logic and architecture-specific JIT implementations.

## KASAN Instrumentation Progress

The KASAN hardening workstream has continued progressing from early prototyping toward upstream integration work.

KASAN (Kernel Address Sanitizer) is a Linux kernel debugging and hardening feature that detects invalid memory accesses such as out-of-bounds reads/writes and use-after-free conditions.

Since the previous update, work has advanced in several areas:

### Expanded JIT Instrumentation Work

Implementation work continued on adding KASAN checks to JIT-generated memory access paths. This included refinement of:
- emitted `__asan_loadX` and `__asan_storeX` call handling;
- stack-access instrumentation logic;
- verifier metadata handling for distinguishing stack and non-stack accesses; and
- runtime handling for poisoned and red-zone memory scenarios.

Additional debugging and testing was also performed around boot-time and runtime stability issues encountered during early implementation.

### Testing and Validation

The proof-of-concept testing environment based on `bpf_testmod` kfuncs was expanded to support additional validation scenarios involving invalid and freed memory references.

This testing continues to help validate instrumentation behavior for JIT-emitted:
- `BPF_LD`
- `BPF_LDX`
- `BPF_ST`
- `BPF_STX`

memory access operations.

### Upstream and Community Engagement

Discussion with upstream maintainers and the broader BPF community continued through RFC discussion and mailing-list review.

Recent work has focused on:
- narrowing the scope of the initial upstreamable implementation;
- improving maintainability of the instrumentation approach;
- refining Kconfig gating and architecture enablement; and
- identifying areas where verifier support may simplify JIT-side logic.

## Current Focus Areas

The current phase of the engagement is focused on:
- continued remediation follow-up for disclosed findings;
- refinement of verifier-related analysis;
- stabilization of JIT KASAN instrumentation behavior;
- improving stack-access handling and instrumentation selectivity; and
- preparing cleaner upstream-ready implementation patches.
