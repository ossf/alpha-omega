## Package Repository Security

The past month on rubygems.org was shaped by a security incident and the work it accelerated. On May 12, a persistent actor created between 1,000 and 2,000 fake accounts and pushed malicious gems that exfiltrated data to external domains, exploiting a gap where unconfirmed accounts could still generate API keys. A second problem surfaced in parallel: the webhook test-fire endpoint was being abused to relay outbound requests and got rate-limited the same day. The broader response took four days: registrations were suspended, disposable and RFC-reserved email domains were blocked at signup, and the Fastly WAF was deployed in logging mode. Rather than push the WAF into blocking mode without understanding the impact on gem install across the ecosystem, we pivoted to Datadog App & API Security in monitor mode, set up live monitoring on auth endpoints, and had registrations back on with new defences in place by end of week. The postmortem is complete, and follow-up is underway: the WAF moves to blocking mode for account endpoints in early June, rate limits for new accounts are in design, and the incident playbook is being updated to include comms protocols.

May also closed some gaps that had been sitting open for years and laid groundwork for the next layer of abuse prevention. API key expiry is now live, with configurable windows from one day to ninety; gem signout now revokes the key server-side rather than just clearing it locally. Together they mean a stolen or forgotten credential has a shorter useful life. The compact index v2 rollout completed production cutover this week, and for the first time created_at timestamps are part of the index. That's the foundation for a cooldown feature, where clients can hold back newly published gems for a configurable window before treating them as installable — buying time for detection before wide distribution. Glasswing held its first full team meeting this week, with the first concrete work stream being trusted publishing API key exposure: identifying Ruby projects still using long-lived rubygems.org API keys instead of OIDC. Andrew already has 10 projects flagged from prior PyPI analysis.

The transparency log RFC is also ready to go public. The proposal adds an append-only public log of gem ownership changes, trusted publishing configuration updates, and admin actions to rubygems.org, modeled on Sigstore/Rekor and queryable via a public REST API. The motivation is concrete: when the rest-client gem was compromised in 2019, a malicious version was live for five days before anyone noticed, and there was no audit trail of the account takeover or the unusual publishing activity, nothing for automated tooling to watch. The RFC is targeted for Q3 2026, built with a Google Summer of Code contributor starting in June. Making that kind of takeover detectable in near-real-time is the goal.

## AI Security Engineers in Residence

The AI Security Engineers in Residence program became official in May with a signed agreement with Alpha-Omega. Prep work from April put us in a strong position to move quickly.

We welcomed several members to join Colby and Marty on the team: Dushan, Matt Mongeau, Patrick Linnane, and Maciej Mensfeld. Marty also continued coordinating with Ruby Core and a potential additional engineer in the US.

The team explored using Scrutineer as a centralized scanning platform. Though we paused that work, open logistical questions remain around coordinating scans across thousands of gems and managing maintainer interactions at scale.

Marty attended the Alpha-Omega roundtable in Minneapolis, joining ecosystem peers for a full day of sessions on the program. The morning focused on how to deal with "bernies", unmaintained projects that appear alive. Scanning efforts will be the straightforward part. There was good discussion on the range of options with how different communities might approach it. The afternoon addressed the realities of operating a "bat phone", supporting maintainers effectively, and avoiding burnout. Marty took extensive notes and returned with many ideas for the team.
With the team formed, we held our inaugural kickoff meeting at the end of the month to align on initial goals and workstreams.  We invited Andrew Nesbit to work closely with us on the program launch. The team discussed how vulnerability scanning was not the biggest challenge, instead effective maintainer coordination and community engagement would be much harder.  Developing the program operating model to ensure gems get patched without burning out their maintainers. Initial priorities include addressing API key exposure in RubyGems trusted publishing, coordinating vulnerability disclosures with the Rails core team, and announcing the program to our community.
