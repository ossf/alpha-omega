## AI Security Engineer in Residence (SEIR)

Thanks to the generous funding by Alpha-Omega, the Rust Foundation is excited to [announce](https://rustfoundation.org/media/an-ai-security-engineer-in-residence-for-the-rust-ecosystem/) that it has hired Jacob Finkelman as its AI Security Engineer in Residence (SEIR). Jacob hit the ground running in June, onboarding into the role with a focus on community engagement, tooling ramp-up, and initial vulnerability scanning across the Rust ecosystem.

**Community Outreach and Collaboration:** Jacob connected with key community members to establish trust, plan collaborative workflows, and help shape the role's direction. Jacob also began coordinating with other Security Engineers in Residence (SEIRs).

**Vulnerability Discovery and Reporting:** Using [Scrutineer](https://github.com/alpha-omega-security/scrutineer), Jacob scanned over 200 crate repositories that are dependencies of Rust itself, focusing on those with unsafe code, build scripts, or proc macros that are not maintained by large organizations. ## More Secure Infra

The Rust Project depends on a broad set of infrastructure. Security work in this environment has to be both effective and practical. We always want to move toward stronger isolation, more redundancy, and smoother maintenance workflows, but we also need improvements that can be deployed with the capacity we have today. That is why we recently adopted [Ubuntu Pro](https://ubuntu.com/pro) across parts of the Rust infrastructure fleet. [Canonical](https://canonical.com/) donated 25 Ubuntu Pro subscriptions to the Rust Foundation and we are grateful for their support. That donation lets us improve the security posture of Rust's infrastructure while also lowering the operational burden for the Infrastructure team by providing broader security patch coverage than standard Ubuntu LTS maintenance, livepatch support for applying eligible kernel security fixes without a reboot and long-term security maintenance for Ubuntu LTS releases for up to 10 years.

We continue to move forward in our [efforts](https://rust-lang.zulipchat.com/#narrow/channel/242791-t-infra/topic/MFA.20with.20hardware.20security.20keys.20for.20critical.20infra.20access/with/595043417) to harden security for critical infrastructure. Documentation on how to use and manage the new 2FA policies in place was written by Ubi and is being [reviewed](https://github.com/rust-lang/rust-forge/pull/1074). A blog post on securing critical Alpha-Omega infrastructure is planned for July.

Marco [wrote](https://github.com/rust-lang/infra-team/pull/269) a policy that explains the requirements external runners must meet to be used in the `rust-lang`
GitHub organization, based on the experiments of running external runners in the org and the discussions we had in the last year.

## Sustainability

We are game-theorying what a sensible rate limiting policy beyond [publishing](https://crates.io/docs/rate-limits) may look like for crates.io. Nothing is official yet and there are multiple ways to approach this, but it has gone from let's figure this out later to actual idea generating about ways it could be implemented if we went that route.

A blog post on how Datadog has been beneficial in our efforts in crates.io sustainability is also forthcoming.

## crates.io

Tobias has made several security-related improvements to crates.io this month:

- Fixed authentication API endpoints to use POST with JSON body instead of GET with query parameters, preventing caching and leaking secrets in request logs. 
- Implemented a `reverse_dependencies` database table to improve API endpoint performance. 
- Implemented a code viewer for each crate version, right within crates.io. Here is an [example](https://crates.io/crates/syn/2.0.118/code/src/lib.rs).

## Security in the Ecosystem
