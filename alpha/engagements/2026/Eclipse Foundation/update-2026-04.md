# Vulnerability Management

### Vulnerability Reporting

During this period, a significant portion of the team's efforts was dedicated to identify and address vulnerabilities within Eclipse Foundation projects' codebases, building on capabilities from frontier models. Systematic scans were conducted across Eclipse Foundation projects, with the goal of uncovering and proactively surfacing potential security issues at scale.

Over 2000 repositories were scanned. Findings are now being reported incrementally to the affected projects, prioritizing them by decreasing order of severity.

### Reporting Infrastructure

Following an expected increased volume of vulnerability reports generated by this initiative, efforts have been directed toward strengthening and consolidating report intake and triage capabilities. This includes a review and refinement of existing processes to improve efficiency in triaging findings and coordinating remediation efforts. In parallel, various automation opportunities are being explored to streamline and enhance the current GitLab-based workflow.

### CVE assignment Process

The current CVE assignment process was further refined through its separation into two distinct stages: CVE reservation and CVE publication. This enables improved coordination with project teams and greater clarity throughout the vulnerability disclosure lifecycle. The Eclipse Project Handbook has been updated to reflect the new process: [https://www.eclipse.org/projects/handbook/\#vulnerability-cve](https://www.eclipse.org/projects/handbook/#vulnerability-cve).

# Incident Management and Remediation Support

In response to Copyfail (CVE-2026-31431), a Linux kernel local privilege escalation vulnerability, prompt action was taken to reduce exposure across the organization's Linux environment. Emergency patching activities were prioritized to ensure timely remediation and limit potential impact.

Following the [Bitwarden CLI compromise](https://socket.dev/blog/bitwarden-cli-compromised#Recommendations), the need for more performant tooling to detect potential account compromise was identified. In response, the team developed dedicated tooling to strengthen detection capabilities:

# Otterdog

### CI/CD Hardening

* **Workflow Validation Fix:** A correction to workflow-settings validation logic was applied, ensuring that CI configuration validation behaves correctly against submitted workflow settings ([PR \#633](https://github.com/eclipse-csi/otterdog/pull/633)).

### Access Control & Policy Governance

* **Team Permissions Management:** First-class support for managing GitHub team permissions was added to the Repository model, introducing a team\_permissions field that makes team–repository access control a declarative, auditable part of the Otterdog configuration. Updates, including additions, removals, and permission level changes, are applied automatically during repository creation and synchronization ([PR \#612](https://github.com/eclipse-csi/otterdog/pull/612)).  
* **Configuration Root Isolation:** A new OTTERDOG\_CONFIG\_ROOT environment variable was introduced, allowing operators to specify a custom root directory for Otterdog configuration files. This supports isolated deployment contexts and reduces the risk of configuration files being resolved from unintended paths ([PR \#632](https://github.com/eclipse-csi/otterdog/pull/632)).

# Open VSX

### Malware & Vulnerability Scanning

* **Scan Dispatcher Race Condition Fix:** A race condition in the scan job dispatcher was identified and resolved, where jobs could be dispatched concurrently but their state was not reflected in the transaction immediately, causing scans to stall in a queued state. This was causing real-world scan timeouts reported by publishers ([PR \#1740](https://github.com/eclipse-openvsx/openvsx/pull/1740), [Issue \#1734](https://github.com/eclipse-openvsx/openvsx/issues/1734)).  
* **Scan Admin Dashboard Improvements:** Scanner deep links and human-readable summary extraction were added to the admin scan dashboard, allowing operators to correlate scan jobs with external scanner results and receive richer threat summaries directly in the audit log ([PR \#1789](https://github.com/eclipse-openvsx/openvsx/pull/1789)).  
* **Icon Format Validation at Publish Time:** A check was added to reject extensions with disallowed icon formats at publish time, closing a long-standing security issue around the use of SVG images that could carry embedded scripts ([PR \#1766](https://github.com/eclipse-openvsx/openvsx/pull/1766), [Issue \#2](https://github.com/eclipse-openvsx/openvsx/issues/2)).  
* **File Resource Integrity Validation:** Synchronous publish-time validation was extended to confirm all referenced file resources are accessible before activating an extension version, preventing dangling extension states caused by missing resources ([PR \#1776](https://github.com/eclipse-openvsx/openvsx/pull/1776)).  
* **Vulnerable Extension Version Removal:** A publisher reported a critical security flaw in all versions of brainsync below v3.0.0 — older versions shipped with an unsecured local MCP server allowing bypassing of commercial licensing controls. The vulnerable versions were removed from the registry ([Issue \#9438](https://github.com/EclipseFdn/open-vsx.org/issues/9438)).

### Rate Limiting & Access Control

* **Rate Limiting Phase 2 Rollout:** The second phase of the rate limiting system was merged and deployed, completing enforcement of API usage tiers across the registry ([PR \#1698](https://github.com/eclipse-openvsx/openvsx/pull/1698), [PR \#1731](https://github.com/eclipse-openvsx/openvsx/pull/1731)).  
* **Rate Limiting Configuration Hardening:** IP address extraction logic for the rate limiter was updated to use the correct header (X-Real-IP from nginx) and logging verbosity for the Redis connection watchdog was reduced to avoid log flooding ([PR \#9354](https://github.com/EclipseFdn/open-vsx.org/pull/9354)).  
* **Free Tier Enforcement:** The website was updated to reflect and communicate free-tier enforcement to users ahead of the cutover, including updated banners and links to commercial usage information ([PR \#9507](https://github.com/EclipseFdn/open-vsx.org/pull/9507)).

### Security Policy & Vulnerability Disclosure

* **Vulnerability Reporting Program Update:** The open-vsx.org/security page was updated to reflect a new vulnerability researcher program, making the security policy available at a canonical URL and updating contact information to the dedicated security@open-vsx.org address throughout the website ([PR \#9716](https://github.com/EclipseFdn/open-vsx.org/pull/9716), [Issue \#9650](https://github.com/EclipseFdn/open-vsx.org/issues/9650)).
