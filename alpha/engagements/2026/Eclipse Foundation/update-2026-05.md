# Vulnerability Management

### Reporting Infrastructure

Building on prior efforts to consolidate report intake and triage capabilities, work is underway on a new automation utility that provisions a dedicated, private vulnerability-reporting space in GitLab for every Eclipse Foundation project registered in the Project Management Infrastructure (PMI). These Gitlab groups offer each project a consistent, standardized venue in which to coordinate on a fix under embargo and collaborate directly with the reporting researcher. 

The utility additionally keeps membership of subgroups synchronized with the corresponding project's designated security team as recorded in the PMI, ensuring that only the project security team members can access and triage incoming reports. This feature drastically decreases operational burden through reducing manual effort previously required for per-project issue assignment.

For reporters, the disclosure process remains unchanged; for committers, the changes are minimal. The updated process will be documented in the coming months.

# Optional Committer Identity Verification

A voluntary identity verification program for Eclipse Foundation committers was finalized this month and prepared for general availability on June 2\. The program offers participating committers an additional means of demonstrating that the name displayed on their account has been verified against a government-issued identity document, providing an account-level trust signal while preserving the open, community-led nature of Foundation projects. **Participation is entirely optional and introduces no changes to the Foundation's mission, governance model, or existing committer requirements**. Verification is performed through iDenfy ([https://idenfy.com](https://idenfy.com)), a European Union–based provider, with a successful check resulting in a verification badge on the committer's profile.

The program was designed around data minimization: government-issued ID images and related source materials are processed exclusively by iDenfy and are not retained by the Eclipse Foundation, which stores only the validation certificate returned upon successful verification (issuance date, validated field values, and a digital signature). iDenfy retains source materials for up to 90 days before deletion per the configured retention policy. 

More details about this can be found in the following blog post: [https://mikael.barbero.tech/blog/post/2026-05-27-optional-identity-verification-ga/](https://mikael.barbero.tech/blog/post/2026-05-27-optional-identity-verification-ga/).

Supporting documentation and a FAQ are being prepared for publication in the Eclipse Project Handbook ([https://www.eclipse.org/projects/handbook/](https://www.eclipse.org/projects/handbook/)) ahead of general availability, with community feedback gathered through the Eclipse CSI GitHub Discussions area ([https://github.com/orgs/eclipse-csi/discussions](https://github.com/orgs/eclipse-csi/discussions)).

# Project Identity Authority (PIA)

### Authentication Hardening

* **Issuer-Aware Workload Model:** The project model was redesigned to replace a flat, opaque configuration structure with typed workload subclasses `GitHubWorkload` and `JenkinsWorkload`. Each subclass enforces issuer-specific claim requirements at the model level: GitHub workloads now require the numeric repository owner ID as a mandatory claim, preventing token reuse across renamed or recycled GitHub accounts. Jenkins workloads are bound to their specific issuer URL. Authentication now fails if more than one workload matches the incoming token claims, eliminating a previously silent ambiguity where the first matching entry was used ([PR \#40](https://github.com/eclipse-csi/pia/pull/40), [Issue \#5](https://github.com/eclipse-csi/pia/issues/5)).  
* **Early OIDC Issuer Rejection:** The preliminary issuer check now validates against a static list of known issuers (with full-match for GitHub and prefix-match for Jenkins) instead of scanning all registered workloads. This reduces the cost of rejecting requests from unknown issuers before any token verification occurs ([PR \#40](https://github.com/eclipse-csi/pia/pull/40), [Issue \#6](https://github.com/eclipse-csi/pia/issues/6)).  
* **Authentication Decoupled from SBOM Upload:** OIDC authentication logic was extracted from the SBOM upload endpoint into a standalone FastAPI dependency function. This separation makes the authentication path independently testable, reusable across future endpoints, and ensures that credential verification is not entangled with business logic ([PR \#41](https://github.com/eclipse-csi/pia/pull/41), [Issue \#7](https://github.com/eclipse-csi/pia/issues/7)).

### Configuration & Authorization Backend

* **PostgreSQL-Backed Workload Registry:** The static `projects.yaml` file used to configure authorized workloads was replaced with a PostgreSQL database backed by SQLAlchemy ORM. Project data can now be updated at runtime without restarting the service, removing a gap where credential or authorization changes required a full deployment cycle ([PR \#40](https://github.com/eclipse-csi/pia/pull/40), [Issue \#29](https://github.com/eclipse-csi/pia/issues/29)).  
* **Database Management CLI:** A `pia` console entry point was added with `add-workload` and `add-dt-project` subcommands, providing a controlled interface for registering GitHub and Jenkins workloads and linking them to their DependencyTrack upload targets ([PR \#45](https://github.com/eclipse-csi/pia/pull/45)).

# Otterdog

### Policy Enforcement Reliability

* **Fork PR Approval Policy for Private Repositories:** Otterdog was hard-failing with a runtime error when attempting to read or update the fork PR contributor approval setting on private repositories, after the GitHub API began returning a 422 for that endpoint on private repos. The fix adds a visibility check before calling the endpoint, matching the same guard already in place elsewhere in the provider layer. This restores Otterdog's ability to manage and enforce fork PR approval policies without disrupting unrelated changes in the same apply run ([PR \#647](https://github.com/eclipse-csi/otterdog/pull/647), [Issue \#635](https://github.com/eclipse-csi/otterdog/issues/635)).  
* **Custom Property Enforcement:** Otterdog was sending `allowed_values: []` for `string` and `true_false` custom property types in GitHub API payloads, which the API rejects with a 422 since `allowed_values` is only valid for `single_select` and `multi_select` types. This caused org custom property creation to silently fail and blocked downstream repository-level property updates. The fix strips `allowed_values` from the provider payload for incompatible types, restoring reliable enforcement of org-level custom properties used for governance and compliance classification ([PR \#653](https://github.com/eclipse-csi/otterdog/pull/653), [Issue \#648](https://github.com/eclipse-csi/otterdog/issues/648)).

### CI/CD Supply Chain

* **Automatic Staging Deployment:** Container images are now published with a `dev` tag on every push to the main branch, enabling automatic updates to the staging environment and reducing the window between a merged fix and its availability for validation. The existing `SNAPSHOT` tags are preserved for backward compatibility ([PR \#622](https://github.com/eclipse-csi/otterdog/pull/622)).

# Open VSX

### Malware & Vulnerability Scanning

* **Zip Normalization Check:** The malicious zip detection service was extended to flag duplicate filenames that appear only after path normalization, closing a variant of zip-slip attacks where two entries resolve to the same path on extraction ([PR \#1807](https://github.com/eclipse-openvsx/openvsx/pull/1807)).  
* **Scan Retry Mechanism:** Failed scanner jobs can now be retried directly from the admin dashboard, resolving a long-standing gap where a transient scanner failure left extensions in an unscanned state with no recovery path ([PR \#1832](https://github.com/eclipse-openvsx/openvsx/pull/1832), [Issue \#1741](https://github.com/eclipse-openvsx/openvsx/issues/1741)). A follow-up fix ensured that all scanner job states are included in the retry query ([PR \#1882](https://github.com/eclipse-openvsx/openvsx/pull/1882)).  
* **Conditional Scan Scheduling:** Recurring scan jobs are now only registered when scanning is actually enabled at startup and are deleted when it is disabled, preventing spurious job accumulation and reducing noise in non-scanning deployments ([PR \#1840](https://github.com/eclipse-openvsx/openvsx/pull/1840)).  
* **YARA Non-Root Execution:** The YARA scanning service was reconfigured to run as non-root user 1001 in the production Kubernetes deployment, reducing the blast radius of a potential compromise of the scanner process ([PR \#10731](https://github.com/EclipseFdn/open-vsx.org/pull/10731)).  
* **YARA Stability Fix:** YARA's `MAX_RECURSION` limit was raised to 16 and resource limits were increased after a deeply nested extension archive caused repeated pod crashes in production, ensuring scan availability is not disrupted by adversarial or malformed inputs ([PR \#10711](https://github.com/EclipseFdn/open-vsx.org/pull/10711)).  
* **Scanner Resource Tuning:** CPU and memory resource request/limit assignments for YARA and ClamAV were corrected (requests and limits had been swapped), and argus scanner concurrency and timeout settings were reduced to prevent resource exhaustion ([PR \#10460](https://github.com/EclipseFdn/open-vsx.org/pull/10460), [PR \#10113](https://github.com/EclipseFdn/open-vsx.org/pull/10113)).

### Authentication & Account Security

* **OAuth2 Account Takeover Prevention:** A check was added to compare the `authId` supplied by the OAuth2 provider at login time against the value stored for the matched user account. A mismatch is now treated as a potential account takeover attempt and login is rejected, mitigating username-recycling attacks on OAuth providers ([PR \#1819](https://github.com/eclipse-openvsx/openvsx/pull/1819)).

### Code Security

* **XXE Protection in Sitemap Parsing:** XML External Entity (XXE) protections were added to the sitemap XML parser, preventing a server-side request forgery or information disclosure attack through a maliciously crafted upstream sitemap response ([PR \#1876](https://github.com/eclipse-openvsx/openvsx/pull/1876)).  
* **Package Manifest Consistency Validation:** At publish time, key metadata fields in `package.json` are now validated against the values in `extension.vsixmanifest`. Discrepancies between the two files are rejected, closing a differential parsing path that could be exploited to present different extension identities to VS Code and Open VSX ([PR \#1825](https://github.com/eclipse-openvsx/openvsx/pull/1825)).

### Infrastructure & CI/CD Hardening

* **Redis TLS and Authentication:** TLS encryption and password authentication were enabled for all Redis connections in the EKS production deployment, ensuring session and cache data is protected in transit within the cluster ([PR \#1879](https://github.com/eclipse-openvsx/openvsx/pull/1879), [PR \#10629](https://github.com/EclipseFdn/open-vsx.org/pull/10629)).  
* **Kubernetes Security Contexts:** Security contexts were added to the main application deployment manifest, restricting pod-level privileges and aligning the deployment with least-privilege container security practices ([PR \#10735](https://github.com/EclipseFdn/open-vsx.org/pull/10735)).  
* **Read-Only Mode:** A dynamic read-only mode was introduced that can be toggled at runtime from the admin dashboard. When active, all mutating API endpoints return a `409` response and recurring write jobs pause, providing a safe maintenance window for database migrations without requiring a full shutdown ([PR \#1835](https://github.com/eclipse-openvsx/openvsx/pull/1835)).  
* **S3 Credential Refresh Fix:** The `S3Presigner` was changed to a singleton to prevent IRSA credential refresh failures on ROSA/EKS deployments. Previously, closing the presigner on every call destroyed the STS connection pool, causing all storage operations to fail after the 63-minute credential expiry window ([PR \#1856](https://github.com/eclipse-openvsx/openvsx/pull/1856)).  
* **CodeQL Scope Reduction:** The CodeQL static analysis workflow was restricted to branch push events only, removing the pull request trigger that was producing a high volume of false positives. The scan remains active for merged code ([PR \#1881](https://github.com/eclipse-openvsx/openvsx/pull/1881)).

### Security Policy

* **Security Reporting Guidelines:** The vulnerability reporting instructions in the `open-vsx.org` repository were revised to include clearer guidance on what information to include when submitting a security issue ([PR \#10253](https://github.com/EclipseFdn/open-vsx.org/pull/10253)).
